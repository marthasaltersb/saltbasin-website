# Deploying Salt Basin Net Works (Render + Netlify Hybrid)

This guide deploys the app with the backend on **Render** and the frontend on **Netlify** (where `saltbasin.net` DNS is already pointing). Netlify proxies `/api/*` and `/uploads/*` to Render so the browser only ever talks to `saltbasin.net` — no CORS, no cross-site cookie issues.

**Order matters.** Deploy the backend first, then the frontend.

---

## Part 1 — Backend on Render (~10 min)

### 1.1 Create the Render service

1. Go to https://dashboard.render.com.
2. **New +** → **Blueprint**.
3. Connect GitHub if asked → pick `marthasaltersb/saltbasin-website`.
4. Render reads `render.yaml` and shows the proposed service. Click **Apply**.

### 1.2 Set the required env vars

When Render prompts (or via the service's **Environment** tab after):

| Variable | Value |
|---|---|
| `ADMIN_EMAIL` | `betsy@saltbasin.net` (or whatever you want for login) |
| `ADMIN_INITIAL_PASSWORD` | A strong password — you'll change it on first login |
| `FRONTEND_ORIGIN` | `https://saltbasin.net` (locks CORS down once Netlify is live) |
| `ANTHROPIC_API_KEY` | Leave blank until the Phase 3 agent is wired |

`SESSION_SECRET` is auto-generated by Render.

### 1.3 Wait for the build, then test directly

Build takes 3–5 min. When the service is green, click into it and copy the URL — it'll look like `https://saltbasin-website.onrender.com`.

Test the backend directly:

- `https://saltbasin-website.onrender.com/api/health` should return `{"ok":true,"ts":...}`
- `https://saltbasin-website.onrender.com/` will also work (the Node server serves the React app too as a fallback) — this is useful for testing before Netlify is wired

**Important:** if Render gave the service a different name than `saltbasin-website`, update the two proxy URLs in `netlify.toml` and push.

## Part 2 — Frontend on Netlify (~10 min)

### 2.1 Connect the repo

1. Go to https://app.netlify.com.
2. **Add new site** → **Import an existing project**.
3. Connect to GitHub → pick `marthasaltersb/saltbasin-website`.
4. Netlify reads `netlify.toml` — confirms build command (`npm install && npm run build`), publish dir (`dist`), and the proxy redirects.
5. Click **Deploy site**.

Build takes 2–3 min. Netlify gives you a temporary URL like `https://<random-name>.netlify.app`.

### 2.2 Point saltbasin.net at this site

If `saltbasin.net` is already configured under a different Netlify site (e.g., the old single-HTML version):

1. Old site → **Domain settings** → remove `saltbasin.net` as a custom domain.
2. New site → **Domain settings** → **Add custom domain** → `saltbasin.net`.
3. Netlify will detect the existing DNS and switch the routing within ~1 min.

If `saltbasin.net` isn't on Netlify yet:

1. New site → **Domain settings** → **Add custom domain** → `saltbasin.net`.
2. Netlify shows you the DNS records to set at your registrar.
3. Set them; HTTPS provisions automatically.

### 2.3 Verify the wiring

Once `saltbasin.net` is live:

- Public site: `https://saltbasin.net/`
- Admin login: `https://saltbasin.net/admin/login`
- API health (proxied through Netlify): `https://saltbasin.net/api/health` should return JSON
- Photo upload: log in, edit the About / Intro section, upload a photo, save, publish, confirm it renders on the public site

If admin login returns 401 with the correct credentials, the issue is almost always the cookie. Check that you're hitting `https://` (not `http://`) and that `FRONTEND_ORIGIN` matches `https://saltbasin.net` exactly in Render's env.

## Free tier caveats

### Render (backend) free tier

- Service **spins down after 15 min of inactivity**. First request after wakeup takes ~30 seconds.
- **Disk is ephemeral.** SQLite data and uploaded photos are wiped on every redeploy or restart.
- The app reseeds default content on boot, so the site is never broken — but your real edits and photos don't persist.

To make data persistent: upgrade Render to **Starter** ($7/mo), then uncomment the `disk:` block at the bottom of `render.yaml` and push. The disk mounts at the data directory so SQLite + uploads survive.

### Netlify (frontend) free tier

- 100 GB bandwidth per month — more than enough for a portfolio site.
- 300 build minutes per month — also fine.
- No caveats that affect this setup.

## Updating the site after deploy

Push to `main` on GitHub. Both Render and Netlify auto-deploy from `main` on every push — backend rebuilds on Render, frontend rebuilds on Netlify, both in parallel.

For content edits (text, photos, social links, page structure), use the admin at `https://saltbasin.net/admin/login` — those don't require redeploys. They're stored in SQLite on Render. (Reminder: free tier = wipes on Render redeploys.)

## Going to production

Before pointing real traffic at this:

1. Change `ADMIN_INITIAL_PASSWORD` on first login.
2. Add the persistent disk on Render (you don't want to lose user data).
3. Enable the pre-launch landing gate in admin while you iterate.
4. Set `FRONTEND_ORIGIN` precisely (must match `https://saltbasin.net` exactly — no trailing slash).
5. Set `ANTHROPIC_API_KEY` once the editor agent / BestyStaff (Phase 3 / Phase 5) is wired.

## Troubleshooting

**Netlify build fails on `npm install`** — almost always a Node version mismatch. Confirm `NODE_VERSION = "22.13.0"` is in `netlify.toml` under `[build.environment]`.

**Site loads at `saltbasin.net` but `/api/health` returns 404** — Netlify proxy isn't pointing at the right Render URL. Edit `netlify.toml`, fix the two `to = "https://..."` lines, commit, push.

**Admin login works at `onrender.com` but fails at `saltbasin.net`** — `FRONTEND_ORIGIN` on Render isn't `https://saltbasin.net`. Fix and redeploy the backend.

**Uploads succeed but the image is broken on the public site** — confirm `/uploads/*` proxy is in `netlify.toml`. If you check Network in devtools, the image URL should be `https://saltbasin.net/uploads/<filename>` (NOT `onrender.com`).

**Cold start frustration** — pay the $7/mo for Render Starter (no spin-down), or use an uptime pinger (UptimeRobot free) to hit the backend every 14 min.
